# Матрица поддержки нод (Node Support Matrix)

Документ фиксирует состояние поддержки **новых пакетных нод** из `@multicode/std` на текущем этапе.

## Что считаем «новыми нодами»

В этой матрице «новыми» считаются ноды, добавленные в стандартный пакет как расширение базового ядра и не имеющие прямых аналогов в `visprog::core::NodeTypes`.

## Матрица статусов по новым нодам

Легенда:
- ✅ — поддержка есть в текущем контуре.
- ⚠️ — частично (например, через fallback/совместимость).
- ❌ — поддержки нет, требуется доработка.

### Ноды, которые пока остаются только на TS-слое

Следующие семейства нод **не считаются direct-core** и пока поддерживаются только на уровне TS:
- расширенный control-flow (например, `WhileLoop`, `Parallel`, `Gate`),
- расширенная математика/сравнения/логика вне `Add`,
- расширенные I/O и служебные ноды,
- пакетные ноды `@multicode/std`, перечисленные в матрице выше.

Для этих нод C++ core-совместимость должна добавляться отдельно (NodeTypes/NodeFactory/GraphSerializer + тесты миграции портов).

## Основания для статусов

1. **Доступность в палитре:**
   - Ноды объявлены в `packages/std/package.json` и загружаются в реестр пакетов через `usePackageRegistry`.
2. **TS codegen:**
   - Для указанных нод в `@multicode/std` отсутствует `codegen.cpp.template`; шаблонный генератор при отсутствии шаблона возвращает пустой результат.
3. **C++ core:**
   - В `include/visprog/core/Types.hpp` и `src/core/NodeFactory.cpp` нет прямых определений/конфигурации для этих типов нод.
4. **Сериализация:**
   - Состояние графа сериализуется как `GraphState` целиком (`serializeGraphState`), а snapshot сохраняет сериализованный граф в base64; для перечисленных нод отдельного запрета нет.

## Миграционные заметки

### 1) Новые типы нод (пакетный слой)

При добавлении новых типов нод в `packages/std/package.json`:

1. Добавить определение ноды в пакет (тип, порты, категория, RU/EN-подписи).
2. Если нужна генерация C++, добавить `codegen.cpp.template` (и `includes/before/after` при необходимости).
3. Проверить поведение в палитре Blueprint-редактора и в fallback-сценариях Classic.
4. Обновить эту матрицу и пометить фактический статус поддержки.

### 2) Изменение формата графа

Если меняется формат графа/снапшота:

1. Инкрементировать версию схемы (`GRAPH_SCHEMA_VERSION`) при несовместимых изменениях.
2. Добавить путь чтения legacy-формата (`parseSerializedGraph`/`parseGraphState`) и явную обработку ошибок.
3. Зафиксировать изменения в документации архитектуры и добавить регрессионные тесты сериализации/десериализации.
4. Проверить, что snapshot-блоки корректно читаются/перезаписываются (`@multicode:snapshot begin/end`).

### 3) Минимальный план безопасной миграции

- Шаг 1: Добавить новую ноду в пакет без удаления старых типов.
- Шаг 2: Включить TS codegen через шаблон и покрыть unit-тестом.
- Шаг 3: При необходимости добавить соответствие в C++ core (`NodeTypes` + `NodeFactory`).
- Шаг 4: Только после этого переводить старые графы на новые типы автоматической миграцией.

## Короткий гайд: какие ноды рекомендованы для C++23-проектов

### Рекомендуемый безопасный минимум (на текущем этапе)

Используйте ноды, у которых есть явная поддержка и в пакете, и в генерации, и в ядре:

- **Поток выполнения:** `Start`, `End`, `Branch`, `ForLoop`.
- **Данные/переменные:** `SetVariable`, `GetVariable`.
- **Базовая арифметика:** `Add`.

### Практические рекомендации под C++23

1. Для стабильной генерации используйте в критических графах только ноды с подтверждённым TS codegen.
2. Новые строковые/коллекционные/конверсионные ноды из этой матрицы подключайте только после добавления `codegen.cpp.template`.
3. Если проект строго завязан на C++ core API, избегайте типов нод, отсутствующих в `NodeTypes`/`NodeFactory`, до появления адаптера.
4. Для production-сборок фиксируйте профиль codegen и сохраняйте snapshot графа рядом с исходником для обратимой миграции.
