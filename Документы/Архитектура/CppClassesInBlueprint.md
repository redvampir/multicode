# Реализация C++ классов в Blueprint-редакторе (план внедрения)

## Цель
Добавить в текущий визуальный Blueprint-редактор полноценную поддержку C++ классов (объявление класса, поля, методы, вызовы, конструкторы), не ломая совместимость с существующими графами и Classic-режимом.

## Что уже готово (подтверждённая база)
1. В ядре уже есть тип `DataType::Class`.
2. В webview-типах уже есть `class` как тип данных и инфраструктура для pointer/reference-метаданных.
3. В сериализации предусмотрены алиасы Blueprint/Classic для ключевых узлов, что упрощает обратную совместимость.
4. В пакетной схеме/типах уже существует `class` в допустимых типах портов.

Это означает, что можно идти эволюционно: добавлять *классовую модель* и *классовые узлы* поверх существующей типовой системы, а не переписывать редактор.

---

## Предлагаемая архитектура

### 1) Новая сущность: Class Registry (в состоянии графа)
Добавить в `BlueprintGraphState` секцию `classes`:

- `id`, `name`, `namespace?`
- `members`: поля (`name`, `dataType`, `typeName?`, `access`, `defaultValue?`)
- `methods`: сигнатуры (`name`, `isConst`, `isStatic`, `returnType`, `params[]`, `access`, `isVirtual?`, `isOverride?`)
- `constructors`: список перегрузок
- `destructor?`

**Почему так:**
- Класс становится отдельным контрактом, а не «магией» в свойствах узлов.
- Упрощается кодогенерация и валидация связей.
- Легче строить миграции схем.

### 2) Узлы для классов (минимальный вертикальный срез)
Добавить узлы первой волны:

- `ClassConstructorCall` — создать/инициализировать объект
- `ClassMethodCall` — вызвать метод
- `GetMember` / `SetMember` — доступ к полям
- `StaticMethodCall` — вызов статического метода

Для `ClassMethodCall` и `Get/SetMember` обязательно поле `target` (экземпляр) и строгая привязка к `classId`.

### 3) Типизация и совместимость портов
- Для `class`-портов `typeName` должен хранить полное имя (`Namespace::Type`).
- Для `pointer<class>` уже существующий `pointerMeta` дополняется `targetClassId` (опционально для миграции).
- В `canDirectlyConnectDataPorts` добавить правило: `class(A)` совместим только с `class(A)` (без неявных кросс-классовых преобразований на старте).

### 4) IPC и валидация (обязательный барьер)
- Новые payload’ы: `class.upsert`, `class.delete`, `class.reorderMember`, `class.reorderMethod`.
- Все IPC-сообщения валидировать через Zod-схемы на входе/выходе.
- Ошибки валидации показывать в UI понятными RU/EN сообщениями.

### 5) Codegen C++
Добавить отдельный этап генерации:

1. Сначала декларации классов (`class ...` / `struct ...`).
2. Затем реализация методов.
3. Затем существующий graph body (функции/глобальная логика).

**Правило:** если граф использует класс, но декларация отсутствует — явная ошибка codegen (не silent fallback).

### 6) UI/UX в Blueprint
- Новая панель: «Классы» рядом с функциями/переменными.
- Создание класса через мастер: имя, namespace, базовые модификаторы.
- Узлы `Method/Member` выбирают класс и элемент через dropdown (как уже сделано для pointer/class bind-паттернов).
- Все строки — через i18n ключи одновременно RU и EN.

## Этапы внедрения (безопасный rollout)

### Этап A — Контракт данных и миграция
- Добавить `classes` в `BlueprintGraphState`.
- Добавить миграцию старых графов (по умолчанию `classes: []`).
- Обновить сериализацию в обе стороны (Blueprint и Classic представления).

### Этап B — Узлы + валидация
- Поддержать новые node types в schema/package loader/factory.
- Добавить Zod-валидацию properties узлов классов.
- Добавить проверки несовместимых подключений.

### Этап C — UI
- Панель классов + редактор сигнатур.
- Встраивание выбора методов/полей в свойства узлов.
- Локализация RU/EN.

### Этап D — Codegen
- Генерация деклараций/реализаций.
- Поддержка вызовов, доступа к полям, конструкторов.
- Диагностика ошибок привязок.

### Этап E — Тесты и hardening
- Unit: типизация, Zod, миграции.
- Integration: создание класса → вызов метода → успешный codegen.
- Регрессия для Classic/Blueprint round-trip.

---

## Критерии приёмки
1. Можно создать класс в UI, добавить поле и метод, и использовать их узлами в графе.
2. Graph сохраняется/загружается без потерь, включая round-trip Blueprint ↔ Classic.
3. Неверные привязки (класс/метод/поле) детектируются валидатором с понятной диагностикой.
4. Codegen выдаёт корректный C++ код (декларации + вызовы + доступ к полям).
5. Все новые UI-строки локализованы на RU и EN.
6. Тестовый набор покрывает минимум: миграцию, IPC-валидацию, codegen, round-trip.

## Риски и как их снять
- **Риск:** разъезд схем между webview и C++ ядром.  
  **Снятие:** единая версия схемы + мигратор + контрактные тесты сериализации.

- **Риск:** поломка Classic-режима.  
  **Снятие:** обратимые конвертеры + интеграционный round-trip тест в CI.

- **Риск:** слишком ранняя поддержка наследования/виртуальности усложнит MVP.  
  **Снятие:** в первой версии оставить inheritance off, но предусмотреть поля модели (`isVirtual`, `isOverride`, `baseClasses[]`) как неактивные.

## Практический MVP (что делать прямо сейчас)
1. Завести `classes` в `blueprintTypes` + миграцию формата.
2. Реализовать один узел `ClassMethodCall` (без overload resolution, только точный выбор сигнатуры).
3. Добавить генерацию вызова `instance.method(args...)`.
4. Написать 3 теста: миграция, IPC-валидация узла, codegen happy-path.

Это даст рабочий сквозной сценарий и основу для последующего расширения (конструкторы, поля, static, const-методы).
