# Blueprint UI: подключаемые библиотеки/фреймворки/файлы и палитра символов

> **Дата:** 2026-02-28  
> **Статус:** Предложение к реализации (MVP → v1)  
> **Цель:** дать пользователю управляемый способ подключать внешние источники кода и автоматически получать узлы по символам (функции, переменные, классы/структуры) в палитре Blueprint.

---

## 1. Что именно нужно решить

### Боль
Сейчас палитра узлов в основном статическая (пакеты/встроенные ноды), и нет явной сквозной модели:
- какие библиотеки/фреймворки подключены к проекту,
- какие файлы к каким графам/функциям привязаны,
- какие символы (функции/переменные/типы) импортированы как доступные узлы,
- почему конкретный узел появился или исчез в палитре.

### Целевое поведение
1. Пользователь добавляет источник: **библиотека**, **фреймворк**, **файл кода**.
2. Система индексирует экспортируемые символы (минимум: имя + аргументы + возвращаемый тип + модификаторы).
3. Символы автоматически превращаются в узлы палитры и видны только в релевантном контексте (подключённые файлы/графы).
4. Есть отдельный режим (как «режим функций»): **граф зависимостей источников**, где видно:
   - файл/граф → какие источники подключены,
   - источник → какие символы реально используются,
   - предупреждения по конфликтам/битым ссылкам.

## 2. Архитектурный дизайн

## 2.1 Новый слой: Source Integration Registry

Добавить единый реестр интеграций (TS-сторона, `vscode-extension/src/shared/` + store webview):

```ts
interface SourceIntegration {
  id: string;
  kind: 'library' | 'framework' | 'file';
  displayName: string;
  version?: string;
  location: {
    type: 'npm' | 'vcpkg' | 'local_file' | 'local_folder' | 'git';
    value: string;
  };
  language: 'cpp' | 'ts' | 'js' | 'mixed';
  status: 'active' | 'indexing' | 'error' | 'disabled';
  diagnostics: IntegrationDiagnostic[];
}

interface SymbolDescriptor {
  id: string;
  integrationId: string;
  symbolKind: 'function' | 'variable' | 'class' | 'struct' | 'method' | 'enum';
  name: string;
  localizedName?: {
    ru?: string;
    source: 'original' | 'user_override' | 'auto_translate';
  };
  namespacePath?: string[];
  signature: {
    args: Array<{ name: string; type: string; optional?: boolean; defaultValue?: string }>;
    returnType?: string;
    isConst?: boolean;
    isStatic?: boolean;
    isAsync?: boolean;
  };
  docs?: { ru?: string; en?: string };
  availability: {
    files: string[];
    functionScopes?: string[];
  };
}
```

### Зачем
- Одна точка истины для палитры, инспектора и codegen.
- Упрощение миграций форматов графа.
- Диагностика и объяснимость («узел недоступен, потому что источник отключён»).
- Поддержка русскоязычного отображения внешних символов без потери оригинальных имён для codegen.

---

## 2.2 Контекстная доступность узлов (важно)

Палитра должна фильтроваться по контексту:
- активный файл/граф,
- активный режим (Blueprint/Classic),
- подключённые источники к текущему графу,
- совместимость типа порта (если вызов из drag-connect).

### Правило видимости
Узел по символу показывается, если:
1. `integration.status === active`,
2. символ разрешён в текущем файле/скоупе,
3. нет блокирующей диагностики (`missing dependency`, `parse error`, `version mismatch`).

---

## 2.3 Слой извлечения символов (Indexer)

Сделать адаптерный индексатор:

```ts
interface SymbolIndexer {
  canHandle(integration: SourceIntegration): boolean;
  index(integration: SourceIntegration): Promise<SymbolDescriptor[]>;
}
```

### Адаптеры MVP
1. **C++ Header Indexer (MVP-lite)**
   - Парсинг сигнатур из заголовков (регулярки + ограниченная грамматика).
   - Поддержать сначала: free functions, простые structs/classes (имена полей), глобальные переменные.
2. **TypeScript Declaration Indexer**
   - Парсинг `.d.ts`/TS AST (через TypeScript Compiler API).
   - Извлечение функций, интерфейсов/классов, экспорта.
3. **Manual JSON Adapter (fallback)**
   - Пользователь может загрузить `symbols.json`, если автоиндексация не сработала.

### Почему так
- Быстрый MVP без полной компиляции C++ проекта.
- Контролируемый риск: у каждого источника есть fallback вручную.

---

## 2.4 Генерация узлов из символов

На основании `SymbolDescriptor` генерируется runtime node-definition (не обязательно писать в package.json):

```ts
interface RuntimeNodeDefinition {
  type: string; // sym:<integrationId>:<symbolId>
  label: string;
  labelRu: string;
  category: 'external' | 'library' | 'framework';
  inputs: PortDef[];
  outputs: PortDef[];
  metadata: {
    integrationId: string;
    symbolId: string;
    sourceSignature: string;
  };
}
```

### Маппинг символов в порты
- `function`: входы = аргументы, выход = return (если не `void`) + `exec` для imperative-контекста.
- `variable`: `Get`/`Set` как два отдельных узла.
- `class/struct`: `Make<Type>`, `Break<Type>`, `GetField`, `SetField` (начать с `Make/Break`).
- `method`: первый вход `self` + аргументы.

---

## 2.5 Отдельный режим: «Карта подключений» (Dependency View)

Новый режим webview (рядом с Blueprint/Classic/Functions):

### Что показывает
- Левый слой: графы/файлы проекта.
- Средний слой: подключённые источники.
- Правый слой: используемые символы (счётчик использований).

### Основные сценарии
- «Покажи, что тянет этот файл».  
- «Какие файлы используют эту библиотеку».  
- «Какие символы неразрешимы после обновления версии».  
- «Какие символы проиндексированы, но ни разу не использованы».

### Технически
- Представление можно сделать на React Flow (единый UI стек с Blueprint).
- Источник данных: registry + graph usage map.
- Клик по символу открывает список нод в графе/переход к месту.

---

## 2.6 Локализация внешних символов (RU overlay)

Для внешних источников нужно разделить **отображаемое имя** и **каноническое имя**:
- `name` — оригинальный идентификатор из источника (используется в генерации кода).
- `localizedName.ru` — пользовательский перевод для UI в Blueprint.

### Модель перевода
1. По умолчанию символ отображается как есть (`source = original`).
2. Пользователь может вручную задать RU-имя (`source = user_override`).
3. Опционально доступен автоперевод как черновик (`source = auto_translate`), но только как предложение до подтверждения пользователем.

### UX-правила
- В ноде и палитре показывать `localizedName.ru`, если оно есть и включён RU-язык.
- В подсказке/tooltip всегда показывать оригинал: `Оригинал: std::filesystem::copy`.
- В инспекторе символа добавить action: `Сбросить перевод к оригиналу`.

### Инварианты
- Локализация не меняет `symbolId`, `type` ноды и сигнатуру портов.
- При реиндексации сохранять `user_override` по ключу `(integrationId, symbolId, signatureHash)`.
- Если сигнатура изменилась и hash не совпал, перевод помечается как `stale` и требует подтверждения.

## 3. Изменения по слоям системы

## 3.1 IPC и валидация (обязательно через Zod)

Добавить новые IPC-сообщения в `shared/messages.ts`:
- `integration/add`
- `integration/remove`
- `integration/reindex`
- `integration/list`
- `integration/diagnostics`
- `symbols/query`
- `dependency-map/get`

Для каждого сообщения:
- `requestSchema` и `responseSchema` через Zod,
- runtime-safe ошибки (`code`, `message`, `details`).

## 3.2 Store webview

В Zustand store добавить слайсы:
- `integrationSlice`
- `symbolCatalogSlice`
- `dependencyMapSlice`
- `symbolLocalizationSlice`

Инварианты:
- состояние индексатора отдельно от UI-состояния,
- optimistic update только для локальных действий,
- rollback при IPC-ошибке.

## 3.3 Кодогенерация

В `codegen/` добавить слой резолва внешних символов:
- маппинг node instance → SymbolDescriptor,
- проверка валидности сигнатуры на этапе генерации,
- авто-сбор include/import зависимостей.

Если символ пропал/изменился:
- узел помечается как `broken` с диагностикой,
- генерация не падает silently, а возвращает контролируемую ошибку.

## 3.4 Совместимость Blueprint + Classic

Даже если новый UX делается в Blueprint, нужно:
- сохранить сериализацию в общем формате графа,
- обеспечить отображение external-узлов в Classic как legacy-блоков,
- добавить migration layer: `graphVersion + integrationBindings`.
- Добавить слой `symbolLocalization`, который опционален для старых графов (мягкая миграция без поломки).

## 4. Пошаговый план внедрения

### Фаза A (MVP, 2–3 спринта)
1. Registry + Zod IPC + базовый UI менеджера интеграций.
2. Indexer для TS (`.d.ts`) + manual JSON.
3. Runtime узлы `function`/`variable` в палитре Blueprint.
4. Базовая Dependency View (файл → источник → символы).

**Критерий готовности MVP:**
- можно подключить минимум 1 локальный файл и 1 TS-библиотеку;
- символы появляются в палитре контекстно;
- кодогенератор корректно собирает импорты;
- есть диагностика битых связей.

### Фаза B (v1)
1. C++ header indexer (ограниченный, но стабильный).
2. Узлы для `class/struct` (`Make/Break`, поля).
3. Обратные ссылки в UI: из ноды к источнику и наоборот.
4. Кэш индексации + инкрементальное обновление.

### Фаза C (hardening)
1. Конфликты имён и namespace aliases.
2. Версионирование интеграций и lock-файл.
3. Метрики: неразрешимые символы, время индексации, покрытие использования.

---

## 5. Риски и как страхуемся

1. **Ложные парсинги C++**  
   Мера: ограниченный поддерживаемый синтаксис + manual JSON fallback.

2. **Падение производительности палитры**  
   Мера: индекс в памяти + debounce поиска + виртуализация списка.

3. **Поломка старых графов**  
   Мера: миграция через `graphVersion`, обратная совместимость узлов как `legacyExternalNode`.

4. **Непрозрачные ошибки пользователю**  
   Мера: единый diagnostics panel с action hints («Переиндексировать», «Открыть источник», «Обновить путь»).

## 6. Минимальный контракт данных для графа

```json
{
  "graphVersion": 4,
  "integrationBindings": [
    {
      "integrationId": "boost_1_87",
      "attachedFiles": ["src/main.cpp"],
      "mode": "explicit"
    }
  ],
  "symbolLocalization": [
    {
      "integrationId": "boost_1_87",
      "symbolId": "filesystem_copy",
      "signatureHash": "sha256:...",
      "localizedNameRu": "Копировать файл"
    }
  ],
  "nodes": [
    {
      "id": "node_42",
      "type": "sym:boost_1_87:filesystem_copy",
      "properties": {
        "integrationId": "boost_1_87",
        "symbolId": "filesystem_copy",
        "signatureHash": "sha256:..."
      }
    }
  ]
}
```

`signatureHash` нужен, чтобы быстро определять «символ изменился после обновления библиотеки».

## 9. Критерии приёмки (acceptance)

- Пользователь может подключить минимум 3 типа источников: библиотека, фреймворк, файл.
- Для функции отображаются минимум: имя, аргументы, возвращаемый тип.
- Символы попадают в палитру только в релевантных файлах/графах.
- В отдельном режиме видно связи «файл ↔ источник ↔ символ».
- При ошибке индексации UI показывает понятную диагностику и действие восстановления.
- Сериализация графа не ломает существующие проекты и открывается в Blueprint + Classic.
- Пользователь может задать RU-имя для внешнего символа, и это не влияет на codegen/сигнатуры.

---

## 10. Резюме

Лучший путь — не «подмешивать» это в палитру точечно, а ввести отдельный реестр интеграций + индексатор символов + режим карты подключений. Тогда:
- палитра становится контекстной и объяснимой,
- кодогенерация получает устойчивые контракты,
- пользователю видно, что и откуда подтягивается,
- а команда получает масштабируемую основу для пакетов и enterprise-библиотек.
