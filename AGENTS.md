## I. Роль и Принципы

Агент — автономный инженер-архитектор. Цель: решать задачи «под ключ», не ломая существующую экосистему.

**Ключевые принципы:**
- Поставляй промышленный код сразу: без новых заглушек (`TODO`, `FIXME`, `pass`) в изменённых строках.
- Не задавай рутинные вопросы вида «продолжать ли дальше»; переходи к реализации.
- Не добавляй преамбулы; переходи сразу к сути.
- Не галлюцинируй: если факт не подтверждён, укажи допущение явно.
- Не ломай существующую архитектуру без обоснования и оценки последствий.
- Меняй только релевантные файлы; не трогай посторонние участки кода.

## IV. Рабочий Цикл

Любой запрос обрабатывается по фазам `PLAN -> IMPLEMENT -> VERIFY -> REFINE`.

### 1. PLAN

- Выделить цель, ограничения, инварианты.
- Найти архитектурные риски и зоны неопределённости.
- Зафиксировать проверяемые критерии приёмки.
- Спроектировать решение (псевдокод/последовательность шагов).

### 2. IMPLEMENT

- Писать финальный код с типизацией и обработкой ошибок.
- Валидировать данные на границах (IPC, ввод пользователя, файлы, внешние API).
- Сохранять совместимость с существующим API, если иное не оговорено явно.
- Для C++: `std::expected` / `Result<T>`, RAII, const correctness, C++20 concepts.
- Для TypeScript: `strict`-подход, Zod на границах, избегать `any`.

### 3. VERIFY

- Минимум один проверочный артефакт: unit-тест, интеграционный тест или воспроизводимый скрипт.
- Проверка граничных случаев: `null/undefined`, пустые коллекции, ошибки сети/IO, неверный формат данных.
- Если затронут редактор: проверить Blueprint и Classic режимы.

### 4. REFINE

- Удалить шум и дубли, проверить SRP (один модуль — одна ответственность).
- Обновить документацию и примеры, если изменился API/поведение.
- Проверить итоговый diff на побочные изменения.

## V. Tech Stack и Зоны Кода

### C++ ядро (`src/`, `include/visprog/`)
- Стандарт: C++20
- Сборка: CMake (>=3.25) + vcpkg
- JSON: nlohmann/json
- Тесты: Catch2 (`tests/`)
- Форматирование: clang-format

### VS Code расширение (`vscode-extension/`)
- Язык: TypeScript 5.9+
- UI: React 18 + React Flow (`@xyflow/react`)
- Стейт: Zustand
- Валидация: Zod
- Сборка: Webpack
- Тесты: Vitest (unit), Mocha (integration)
- Линтинг: ESLint
- Legacy: Cytoscape (Classic редактор поддерживается параллельно)

### Структура проекта

```text
MultiCode_VS/
├── src/                    # C++ ядро (граф, узлы, порты)
│   ├── core/               # Graph, Node, Port, NodeFactory, Types
│   └── generators/         # CppCodeGenerator
├── include/visprog/        # C++ заголовки
├── tests/                  # C++ тесты
├── vscode-extension/       # VS Code расширение
│   └── src/
│       ├── extension.ts    # Точка входа
│       ├── webview/        # React UI (BlueprintEditor, nodes/, store.ts)
│       ├── codegen/        # Генерация кода (TS-сторона)
│       ├── panel/          # GraphPanel, переводчик MarianMT
│       └── shared/         # Типы, валидация, сериализация
├── schemas/                # JSON-схемы пакетов и узлов
├── packages/std/           # Стандартная библиотека узлов
└── Документы/              # Архитектура, статусы, отчёты
```

## VI. Обязательные Инварианты

1. **Версия релиза** берётся только из `vscode-extension/package.json`.
2. **Локализация обязательна:** любые новые UI-строки добавляются через систему переводов в RU и EN одновременно.
3. **IPC строго валидируется:** вход/выход сообщений проходит Zod-парсинг на границе.
4. **Оба редактора проверяются:** Blueprint (основной) и Classic (legacy), если затронут редактор или формат графа.
5. **Конвертеры форматов не ломаются:** изменения графа учитывают миграцию между старыми и новыми представлениями.
6. **Документация обновляется** при изменении API, протокола сообщений или добавлении модуля.
7. **Ломающее изменение** должно сопровождаться планом миграции (schema/version notes).

## VII. Стандарты Кода

### C++

- Именование: `PascalCase` (типы), `camelCase` (методы/функции), `snake_case` (переменные/поля), `UPPER_SNAKE` (константы), `snake_case` (namespace).
- Namespace: `visprog::core`, `visprog::codegen`.
- Владение: `unique_ptr` по умолчанию, `shared_ptr` только при разделяемом владении, `weak_ptr` для наблюдателей.
- Raw `new`/`delete` запрещены.
- Строки: `std::string_view` для параметров, `std::string` для владения.
- Ошибки: `std::expected<T, Error>` для ожидаемых ошибок; исключения только для действительно исключительных ситуаций.
- Атрибуты и квалификаторы: `[[nodiscard]]`, `noexcept`, `constexpr` — где корректно.
- Концепты C++20 предпочтительнее SFINAE.
- Strong types для идентификаторов (`NodeId`, `ConnectionId`), не использовать «голые» числа.
- Trailing return type: `auto foo() -> ReturnType`.
- Целевые флаги предупреждений: `-Wall -Wextra -Wpedantic -Werror -Wconversion -Wsign-conversion` (или эквивалент MSVC).

### TypeScript

- Строгая типизация: `strict: true`, избегать `any`.
- Валидация границ: Zod-схемы для IPC и внешних данных.
- React: функциональные компоненты, хуки, `memo`/`useMemo`/`useCallback` для тяжёлых узлов.
- Стейт: Zustand store (`vscode-extension/src/webview/store.ts`).
- Именование: `PascalCase` (типы/компоненты), `camelCase` (функции/переменные), `UPPER_SNAKE` (константы).
- Новые UI-строки не хардкодить в JSX/TS: только через i18n-слой.

### Общее

- SOLID: модуль решает одну ответственность.
- Комментарии объясняют **зачем**, а не **что делает строка**.
- Defensive programming: валидация и явные ошибки на границах системы.
- Если изменён публичный контракт, синхронизируй код, схемы и документацию в одном изменении.

## IX. Definition of Done (DoD)

Задача считается завершённой только если:

- Изменение реализовано полностью, без новых заглушек в затронутом коде.
- Все обязательные проверки (сборка/тесты/линт) выполнены или явно объяснено, почему не выполнены.
- Для новых UI-строк добавлены RU и EN переводы.
- Для изменений IPC есть Zod-схема и обработка ошибок в рантайме.
- Для изменений редактора проверены Blueprint и Classic режимы.
- Документация обновлена при изменении API/контракта/поведения.

## XI. Формат Ответа Агента

### 1. План решения
- Что меняется
- Критерии приёмки

### 2. Реализация
- Полные файлы или чёткие дифы
- Комментарии только в сложных местах (Why, not What)

### 3. Верификация
- Что запускалось (команды)
- Что получено (результат)
- Какие ограничения/риски остались
