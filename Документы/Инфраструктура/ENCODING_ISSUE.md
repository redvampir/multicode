# Проблема с Кодировкой в Windows

**Дата обнаружения:** 16 декабря 2025  
**Статус:** ✅ Решено + Защита реализована

---

## 🔴 Проблема

При попытке сборки проекта на Windows возникала ошибка:

```
CMake Error: The source directory "F:/MultiCode/╨Ь╤Г╨╗╤М╤В╨╕╨Ъ╨╛╨┤" does not exist.
```

**Корневая причина:**  
CMake закэшировал путь проекта с кодировкой Windows-1251 вместо UTF-8, из-за чего кириллические символы превратились в кракозябры.

---

## 🔍 Почему Это Происходит?

### Виновники (в порядке убывания вины):

1. **Windows** (90% вины)
   - Использует CP-1251 для консоли по умолчанию
   - PowerShell до версии 7 не гарантирует UTF-8
   - Системные API возвращают пути в ANSI, а не Unicode

2. **CMake** (5% вины)
   - Кэширует пути при первой конфигурации
   - Не переконфигурируется автоматически при смене кодировки
   - `CMakeCache.txt` сохраняет пути "как есть"

3. **MSVC** (3% вины)
   - Компилятор ожидает UTF-8 BOM для исходников
   - Линкер может ломаться на кириллических путях в PDB

4. **VS Code** (2% вины)
   - По умолчанию не форсит UTF-8 для терминалов
   - Запускает PowerShell 5.1 вместо 7.x
   - Не предупреждает о проблемах с кодировкой

5. **Разработчик** (0% вины, но всё равно страдает)
   - Использует кириллицу в путях, потому что это родной язык
   - Ожидает, что в 2025 году Unicode должен работать везде
   - Наивно полагает, что инструменты умеют работать с не-ASCII символами

---

## ✅ Решение

### 1. Автоматический Скрипт Сборки

Создан [`scripts/build-cmake-utf8.ps1`](../../scripts/build-cmake-utf8.ps1):

```powershell
# Принудительная установка UTF-8
[Console]::OutputEncoding = [System.Text.Encoding]::UTF8
[Console]::InputEncoding = [System.Text.Encoding]::UTF8
$env:PYTHONUTF8 = "1"
$env:LC_ALL = "C.UTF-8"
```

**Возможности:**
- ✅ Автоматическая настройка UTF-8
- ✅ Проверка пути на кириллицу с предупреждением
- ✅ Очистка build директории (`-Clean`)
- ✅ Запуск тестов (`-RunTests`)
- ✅ Выбор конфигурации (`-Configuration Release|Debug`)

**Использование:**
```powershell
# Обычная сборка
.\scripts\build-cmake-utf8.ps1

# Полная очистка и пересборка с тестами
.\scripts\build-cmake-utf8.ps1 -Clean -RunTests

# Debug конфигурация
.\scripts\build-cmake-utf8.ps1 -Configuration Debug
```

### 2. Обновлённые VS Code Tasks

Все таски в [`.vscode/tasks.json`](../../.vscode/tasks.json) теперь используют защищённый скрипт:

```json
{
  "label": "🏗️ Сборка C++ (Release)",
  "type": "shell",
  "command": "powershell",
  "args": [
    "-ExecutionPolicy", "Bypass",
    "-File", "${workspaceFolder}/scripts/build-cmake-utf8.ps1",
    "-Configuration", "Release"
  ]
}
```

### 3. Защита в CI/CD

GitHub Actions (`.github/workflows/cpp-build.yml`) обновлён:

```yaml
- name: Configure CMake
  run: |
    # Принудительная установка UTF-8
    [Console]::OutputEncoding = [System.Text.Encoding]::UTF8
    $env:PYTHONUTF8 = "1"
    
    cmake -B build -S . ...
```

## 📊 Почему НЕ Виноват VS Code?

VS Code — это **текстовый редактор**, а не **система сборки**.

| Компонент | Ответственность | Кодировка |
|-----------|----------------|-----------|
| **VS Code** | Редактирование файлов | ✅ UTF-8 (всегда) |
| **PowerShell** | Выполнение команд | ❌ CP-1251 (по умолчанию) |
| **CMake** | Генерация build-системы | ⚠️ Зависит от консоли |
| **MSVC** | Компиляция C++ | ⚠️ Ожидает UTF-8 BOM |
| **Windows** | Файловая система | ✅ Unicode (NTFS) |
| **Windows API** | Системные вызовы | ❌ ANSI (старое API) |

**Вывод:** VS Code просто запускает PowerShell, который запускает CMake, который запускает MSVC. Каждый из них имеет свои представления о кодировке.

## 🛡️ Как Теперь Защищены?

### ✅ Локальная разработка
- Все таски VS Code используют защищённый скрипт
- Автоматическое предупреждение при кириллице в пути
- UTF-8 форсится на уровне PowerShell

### ✅ CI/CD
- GitHub Actions явно устанавливает UTF-8
- Раннеры используют английские пути (нет проблемы)

### ✅ Новые разработчики
- Документация предупреждает о проблеме
- Скрипт автоматически настраивает окружение
- Понятные сообщения об ошибках

## 🎓 Уроки

### Что работает:
1. ✅ Автоматизация решает проблемы раз и навсегда
2. ✅ Явная настройка кодировки лучше, чем надежда на "по умолчанию"
3. ✅ Проверка предусловий экономит часы отладки

### Что НЕ работает:
1. ❌ Надежда, что "в 2025 UTF-8 везде"
2. ❌ Доверие к настройкам Windows по умолчанию
3. ❌ Игнорирование предупреждений CMake

### Альтернативы (если проблема повторится):
1. Переименовать директорию проекта на английский путь
2. Использовать WSL2 (там UTF-8 работает из коробки)
3. Установить PowerShell 7.x (лучше поддержка UTF-8)
4. Использовать Ninja вместо MSBuild (меньше проблем с кодировкой)
